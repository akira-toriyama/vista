# glossary

vista で使う用語集。ユーザーと Claude Code の認識ズレ防止が目的。
用語の追加・改名はコード変更と同一 PR で行う。

| 用語                                    | 意味                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **board（ボード）**                     | 1 つの `.furrow/` ストア。vista v1 は中央ボード（projects checkout）固定。語彙（lanes/types 等）は `furrow board --json` = `BoardInfo` が正で、ハードコードしない                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **lane（レーン）**                      | タスクの状態列（inbox/backlog/ready/…）。`config.toml [lanes].order` 由来の文字列。TS では `Lane` 型（= string alias）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **shard（シャード）**                   | `.furrow/tasks/<id>.json` 1 ファイル = 1 タスクのメタデータ。TS 型は `furrow schema task` から codegen した `FurrowTaskShardV2`（別名 `TaskShard`）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **codegen**                             | `pnpm codegen`（scripts/codegen-furrow-types.mjs）。`furrow schema task` の JSON Schema から `src/domain/generated/` に TS 型とスキーマ snapshot を生成・コミットする。手書きの Task 型は持たない                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **schema drift guard**                  | contract test の 1 本。live の `furrow schema task` と vendored snapshot を diff し、furrow 側のスキーマ変更で codegen が古びたら CI を落とす                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Task / TaskDetail**                   | `Task` = `furrow ls --json` の 1 行（shard + furrow が計算する `actionable`/`blocked_by`/`container`/`stuck`）。`TaskDetail` = `furrow show --json`（shard + `body_text`）。計算フラグを vista 側で再導出しない（ロジック正本 = furrow）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **MutationReport**                      | `add` 以外の mutation が返す `{after, before, changed}`。`add` だけは before が存在しないため素の shard を返す                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **FurrowPort**                          | application 層の interface。furrow への唯一の出入口で、CLI/JSON 契約の薄い型付きミラー。`sync` は sync 統合タスク到着まで意図的に不在                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **Board view**                          | kanban ビュー（`src/ui/views/board/`）。列 = lane（`BoardInfo.lanes` 由来、ハードコードしない）、列内順序 = sparse priority 昇順（小さいほど上）。UI の北極星 = GitHub Projects                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **placement / Placement**               | タスクの置き場所指定。`{before: id}` / `{after: id}`（相対、furrow が priority を計算・gap 枯渇時は同一 write で respace）または `{priority: n}`（絶対）。相対 2 種は domain の `RelativePlacement`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **drop plan（DropPlan）**               | drag&drop 1 回を furrow の 1 write に写像する domain 純ロジック `planDrop` の出力。同一 lane 内 = `reorder --before/--after`、lane 跨ぎ = `set -s <lane> --before/--after`（lane+位置を 1 write）、空列へ = `move`。位置が変わらない drop は `null`（no write）                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **optimistic update（board）**          | `useDropTask` の onMutate で list cache の status/priority を先に書き換え、カードを即座に新位置へ（スピナー無し）。priority は `optimisticPriority` の暫定値（小数可）で、settle 後の refetch で furrow の正準値に置換。エラー時は snapshot へ rollback                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **blocked 旗**                          | `Task.blocked_by`（furrow 計算）が非空のカードに付く旗アイコン + dim 表示。blocker の id を aria-label で持つ                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **display options**                     | Board card の表示項目 toggle（ID / Pips / Labels / Repo）。`CardDisplayOptions` 型、BoardView ローカル state                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **renumbered**                          | 相対 placement で sparse gap が枯渇し lane 全体が respace された時だけ MutationReport に付く `{id, from, to}[]`（furrow が同一 write 内で atomic に実施）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **furrow-client**                       | `createFurrowClient(exec)`。FurrowPort の exec 非依存コア: argv 組み立て・exit code 契約（0/1/2/3+）の解釈・stderr `{"error":{…}}` 封筒の parse。本番は Tauri invoke、contract test は child_process から同一コードを通す                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **TauriFurrowAdapter**                  | `createTauriFurrowAdapter()`。本番の FurrowPort 実装。`furrow_exec` invoke + `tasks://changed` 購読。runtime で `@tauri-apps/*` に触る唯一のモジュール                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **FurrowError**                         | furrow 境界の typed error。`kind`: `not-found`(exit 1) / `validation`(exit 2) / `internal`(exit 3+) / `core`(Rust CoreError: spawn 失敗等) / `bad-output`(exit 0 だが stdout が期待 JSON でない)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| **exec bridge**                         | Rust 側 `furrow_exec` コマンド。ホストの furrow をボード root cwd で spawn し、生の `ExecResult {code, stdout, stderr}` を返すだけ（解釈は TS 側）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **tasks://changed**                     | Rust の fs-watcher が `.furrow/` 変更で emit する Tauri event。`useTasksChangedInvalidation` が受けて `['tasks']` サブツリーと board を invalidate（CLI からの並行編集に GUI が追随する仕組み）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **queryKey 階層**                       | TanStack Query のキー設計。`['tasks']` を根に `list`/`detail`/`deps` がぶら下がり、書き込み後は根ごと invalidate（どの write も他行の actionable/blocked を変えうるため）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **contract test**                       | `tests/contract/`（vitest の node project）。temp dir に `furrow init` した使い捨てボードへ、本番と同じ furrow-client で実バイナリを叩き JSON 契約を検証。furrow が PATH に無ければ skip                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| **composition root**                    | `src/Main.tsx`。具象 adapter と port が出会う唯一の場所（QueryClientProvider + FurrowPortProvider の配線）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **React Compiler**                      | babel-plugin-react-compiler@1.0.0（exact pin）。vite.config と vitest.config の両方に配線し、テストも compiled 出力を通す。memo 化は compiler の仕事 —— 手書き `useMemo`/`useCallback`/`React.memo` は原則禁止（escape hatch が要る時は理由コメント必須）                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **ref-as-prop**                         | React 19 標準の ref 受け渡し。`forwardRef` は新規に書かない（既存 vendored コードは除く）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **presenter/hook 分離（注入パターン）** | view の家風（t-rz61、範式 = akira-toriyama/next.js-tpl）。1 view = 1 ディレクトリ `Xxx/`: `Xxx.tsx`（presenter `XxxComponent` = 全 state/handler を `props.` で読む純 render + 合成 `Xxx`。component は `function` 宣言・`React.FC` 不可）・`useXxx.ts`（`useXxx(props: OuterProps): Props`、`props.` で読み挙動を集約）・`Xxx.type.ts`（`OuterProps` = 親が渡す公開 API / `Props` = 注入後）・`Xxx.mock.ts`（`makeXxxProps({type: "…"})` factory）。合成は `<XxxComponent {...useXxx(props)} />` の 1 行だけで、そこだけ `/* c8 ignore */` を許す。props が空の view（App）は type/mock を省略してよい                                                                                                                             |
| **coverage 100% ゲート**                | `check:test` = `vitest run --coverage`（v8 provider・include `src/**` 明示・thresholds 4 指標 100）。100% は目標でなく注入パターンの自然な帰結。exclude は curated: `src/domain/generated/`（codegen）・`src/ui/primitives/`（vendored）・`src/Main.tsx`（composition root）・`*.type.ts`・`*.mock.ts`・`*.test-d.ts`。exclude 追加の基準 =「機械生成 or 型のみ or テスト用 fixture or 配線のみ」— 挙動を持つコードは exclude しない                                                                                                                                                                                                                                                                                                |
| **strictRender**                        | presenter テストのヘルパ。`<StrictMode>` で包んで render/rerender し、二重レンダーで React Compiler の memo cache-hit 経路まで踏む（presenter は純 render なので副作用なし）。hook のテストは登録回数を数えるため StrictMode を使わず、「同一 props 再レンダーで再登録しない」テストで hit 経路を踏む                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **primitives**                          | `src/ui/primitives/` = vendored shadcn/ui コンポーネント置き場（旧 `src/ui/components/ui/`）。基盤 = **Base UI**（`@base-ui/react`、style = `base-nova`。選定 = t-wf4p / 移行 = t-614w）。coverage・一部 lint・**knip の対象外**（vendored = 使う前に置かれるので `knip.json` の `ignore` が要る）。手書きせず `pnpm dlx shadcn@latest add <部品>` で再生成する                                                                                                                                                                                                                                                                                                                                                                     |
| **base-nova**                           | shadcn registry の style 名 = Base UI 版（`components.json` の `"style"`。旧 `radix-nova`）。部品名は registry 準拠で **`dropdown-menu`**（`menu` は 404）。radix-nova とは cva（見た目）がバイト同一で、違うのは配線だけ。registry の raw JSON には `IconPlaceholder`（`@/app/…` の authoring import）が入っているが、**CLI が `iconLibrary: "lucide"` で書き換えて出力するので手直しは不要** —— raw JSON を読んで「import が漏れている」と誤読しないこと（`--dry-run --view` も書き換え後を見せる）                                                                                                                                                                                                                               |
| **render prop**                         | Base UI の合成 escape hatch（Radix の `asChild` 相当。`asChild` は Base UI に無い）。`render={<Button />}` に要素を渡すか `(props, state) => ReactElement` を渡す。範例 = `dialog.tsx` の close ボタン（`<DialogPrimitive.Close render={<Button variant="ghost" />}>`）                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **modal 三状態**                        | Base UI Dialog の `modal` は `true` / `false` / `'trap-focus'` の 3 値。**3 箇所が別々の厳密さで読む**（@base-ui/react 1.6.0 の dist 実測）: scroll lock = `modal === true`（`dialog/root/useDialogRoot.js:89`）／ pointer 遮断の InternalBackdrop = `modal === true`（`dialog/portal/DialogPortal.js:43`）／ focus trap = `modal !== false`（`dialog/popup/DialogPopup.js:112`）。背景を触れるまま focus だけ trap する side-peek = `'trap-focus'`。ただし ⚠️ 下記の aria-hidden 事象を伴う                                                                                                                                                                                                                                        |
| **罠: trap-focus の aria-hidden**       | `modal="trap-focus"` でも背景（board）は **AT から消える**。`DialogPopup.js:112` が `'trap-focus'` を `modal={true}` に潰して FloatingFocusManager に渡し、`FloatingFocusManager.js:347` の `markOthers(…, {ariaHidden: modal})` が発火するため。**見えて・スクロールできて・クリックできるのに VoiceOver からは存在しない**（実 WKWebView の AX dump で 0/48 を確認 = t-wf4p §2）。side-peek 実装時に上流 issue か迂回を要検討                                                                                                                                                                                                                                                                                                     |
| **罠: className-as-function**           | Base UI の part は `className` に `(state) => string` を許す。**cva 全廃（2026-07-17）で「cva→clsx が関数を落とす」旧罠は解消**（button.tsx は `(state) => cn(...)` で合成）。原則は不変 = state 依存の class は `data-*` 属性 + Tailwind variant（`data-open:` `data-disabled:` 等・Radix の `data-[state=open]:` は Base UI で効かない）で書く                                                                                                                                                                                                                                                                                                                                                                                    |
| **cva 不使用（判別 union）**            | own-code は cva / tailwind-variants を使わない（2026-07-17 全廃・button.tsx も de-cva 済み）。制約のある variant は **TS 判別 union 型の公開 props + Record マップ + `cn()`** で書く。cva/tv は独立軸で型付けするため不正な組合せ（例「`size=lg` は `intent=primary` の時だけ」の cross-axis 制約）を型で禁止できず、判別 union が唯一解。理由の正典 = projects **t-ww01**。eslint `no-restricted-imports` が cva/tv の import を `src/**` 全域で禁止（`shadcn add` が cva 入り部品を吐いたら手 de-cva）                                                                                                                                                                                                                            |
| **class 文字列は lint 担保**            | 型で守れるのは variant の**選択**（判別 union → **cva 不使用**）まで。class **文字列**そのもの（`bg-blu-500` の typo・並び順）は 2026 でも tsc で型化できない —— 文字列リテラルの中身は tsc に不透明で、Tailwind 公式 typegen も未出荷（tailwindlabs#5095/#8112）。型化を謳う Panda/vanilla-extract/StyleX は Tailwind の"置換"＝vendored primitives を `shadcn add` 再生成から fork させるため不採用。→ 文字列の安全網は型でなく **lint**（**better-tailwindcss**）。天井の正典 = projects **t-04yw**                                                                                                                                                                                                                              |
| **better-tailwindcss**                  | `eslint-plugin-better-tailwindcss`（v4・Tailwind v4 ネイティブ）。`settings["better-tailwindcss"].entryPoint = src/ui/index.css` で実 @theme を解決 → custom token（`bg-primary` 等）・arbitrary variant（`aria-invalid:` `[&_svg…]` 等）を誤検知しない。**no-unknown-classes=error** が typo/未登録 class を全面で弾く: className・`cn()` 引数・**button.tsx の `Record<Variant,string>` マップ**（`selectors` に `{kind:"variable", match:"objectValues"}` を足す＝prettier の届かない面）。**enforce-consistent-class-order=error** は Record マップ**のみ**（JSX+cn リテラルの整列は **prettier-plugin-tailwindcss** が担当＝面分割・両者とも official 順で不衝突）。cva は `.prettierrc` の `tailwindFunctions` からも撤去済み |
| **primitives の投資方針**               | `src/ui/primitives/**` は **vendored の薄層**（Base UI＝機能 ＋ Tailwind＝見た目 の薄い被せ）のまま据え置く。**機能≫見た目**（見た目は good-enough で可）。class の typo/並びは **lint**（typo=better-tailwindcss 全面／並び=prettier-plugin-tailwindcss、Record マップだけ better-tailwindcss。分担は↑「better-tailwindcss」行。cheap・silent 崩れを弾く）で担保するが、**styles.ts 化・全 primitives の dead-code・cn 統一は"見た目層の作り込み"なのでやらない**（投資対効果が薄い）。素材の共通化は将来の**共有 web UI lib（[weft](https://github.com/akira-toriyama/weft)）**＝Sill(Swift) の web 版で行う（vista 単体では早い＝2+ repo 共有時が損益分岐）                                                                      |
| **型テスト（\*.test-d.ts）**            | `expectTypeOf` で codegen 型の形状を固定（vitest typecheck、unit project で tsc 実行）。schema drift guard（contract・要 live furrow）のオフライン補完。範例 = `src/domain/task.test-d.ts`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **PBT（property-based testing）**       | domain 純関数の不変条件は `@fast-check/vitest`（`test.prop`）で検証する。適用基準 =「example で列挙しきれない入力空間 × 実装から独立に書ける不変条件（oracle）を持つ純関数」。範例 = `kanban.property.test.ts`（planDrop の null⇔順序不変 iff）。失敗時は fast-check が出力する seed で再現                                                                                                                                                                                                                                                                                                                                                                                                                                         |
